# game-council — 기획 위원회 토론 스킬

> **Phase:** 제한 없음 (기획 논의가 필요할 때 언제든 호출)
> **입력:** `docs/game-plan.md` (필수), `docs/art-direction.md` (있으면 참조)
> **출력:** `docs/council/game-YYYYMMDD.md`

---

## 역할

4명의 기획 전문가가 현재 기획서를 기반으로 각자의 관점에서 검토하고, 발견/우려/제안을 정리한 뒤 종합 액션 아이템을 도출한다.

---

## 에이전트 구성

### 1. 시스템 기획자
**관점:** 게임이 구조적으로 동작하는가?
- 코어 루프 완성도 (입력 → 피드백 → 보상 → 반복)
- 시스템 간 연결 (고아 시스템 여부)
- 밸런스 수식/곡선의 일관성
- 경제 시스템 (인플레이션, 싱크/소스 균형)
- 난이도 커브 (초반 학습 → 중반 몰입 → 후반 도전)
- 기획 vs 구현 정합성 (코드가 있는 경우: 기획서 수치와 GameConfig 수치 대조)

### 2. 콘텐츠 기획자
**관점:** 플레이어가 할 것이 충분한가?
- 스테이지/에피소드 볼륨 적정성
- 보상 구조 (노력 대비 보상 체감)
- 콘텐츠 소비 속도 vs 생산 비용
- 반복 플레이 동기 (일일/주간 목표, 시즌)
- 엔드콘텐츠 또는 무한 모드 설계

### 3. BM 전문가
**관점:** 이 게임으로 수익을 낼 수 있는가?
- 수익 모델 (광고/인앱/구독) 적합성
- 과금 포인트 자연스러움 (페이월 vs 편의 과금)
- 리텐션 구조 (D1/D7/D30 기여 요소)
- 광고 배치 타이밍 (보상형/전면/배너)
- LTV vs CAC 예상 구조

### 4. UX 기획자
**관점:** 사용자가 막히지 않고 플레이할 수 있는가?
- 첫 경험 (FTUE) 흐름
- 튜토리얼 필요 여부와 방식
- 화면 흐름 (씬 전환 경로)
- 조작 직관성 (터치/키보드)
- 정보 과부하 여부 (HUD, 팝업)
- 접근성 (폰트 크기, 색각 이상 대응)

## 실행 프로세스

### Step 1: 기획서 + 코드 로드

1. `docs/game-plan.md`를 읽는다 (필수, 없으면 중단).
2. `docs/art-direction.md`가 있으면 함께 읽는다.
3. 이전 위원회 기록이 `docs/council/`에 있으면 최신 1개를 참조한다.
4. `src/`에 코드가 있으면 (Phase 3 이후) 설정/밸런스 파일을 읽는다:
   - `src/config/GameConfig.ts` 또는 `src/config.ts` — 밸런스 수치
   - `src/scenes/` 또는 `src/main.ts` — 코어 루프 구현
   - 기획서 수치와 구현 수치의 정합성을 시스템 기획자가 대조한다.

### Step 2: 각 에이전트 발언

4명의 에이전트가 **순서대로** 기획서를 검토하고 발언한다.

각 에이전트는 다음 형식으로 발언:

```markdown
### [역할명] 검토 의견

**발견 (Findings):**
- (기획서에서 확인한 사실)

**우려 (Concerns):**
- (잠재적 문제점, 리스크)

**제안 (Suggestions):**
- (구체적 개선안)
```

### Step 3: 교차 토론

에이전트 간 의견이 충돌하는 부분이 있으면 교차 토론을 진행한다.
예: 시스템 기획자가 "난이도를 높여야 한다" vs UX 기획자가 "진입장벽이 높다"

### Step 4: 종합 액션 아이템

모든 발언을 종합하여 우선순위가 포함된 액션 아이템 목록을 작성한다.

```markdown
## 종합 액션 아이템

| 우선순위 | 카테고리 | 액션 | 제안자 | 근거 |
|---------|---------|------|--------|------|
| P0 (필수) | 시스템 | ... | 시스템 기획자 | ... |
| P1 (권장) | BM | ... | BM 전문가 | ... |
| P2 (선택) | UX | ... | UX 기획자 | ... |
```

### Step 5: 결과 저장 및 보고

1. `docs/council/` 폴더를 생성한다 (없으면).
2. `docs/council/game-YYYYMMDD.md`로 저장한다.
3. 디렉터에게 핵심 요약을 보고한다:

```
"기획 위원회 검토 완료:

🔍 발견: [N]건
⚠️ 우려: [M]건 (P0: [x]건, P1: [y]건)
💡 제안: [K]건

상세: docs/council/game-YYYYMMDD.md

액션 아이템 중 즉시 반영할 항목이 있으면 말씀해주세요."
```

```markdown

> 참석: 시스템 기획자, 콘텐츠 기획자, BM 전문가, UX 기획자

---

## 1. 시스템 기획자 검토

(발견/우려/제안)

## 3. BM 전문가 검토
```

---

## 주의사항

- 이 스킬은 Phase에 종속되지 않는다. Phase 1 기획 초안, Phase 3 중간, 수정 모드 등 언제든 호출 가능.
- 위원회는 **의사결정을 하지 않는다.** 발견/우려/제안만 하고, 최종 판단은 디렉터가 한다.
- 기존 game-validator와의 차이: validator는 기획서의 구조적 모순/누락을 기계적으로 점검하고, council은 각 전문 관점에서 정성적 검토를 수행한다.
