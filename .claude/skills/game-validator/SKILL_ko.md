# game-validator.md — 게임 기획 검증 스킬

> **Phase:** 2
> **입력:** `docs/game-plan.md`, `docs/art-direction.md`
> **출력:** `docs/validation-report.md`

---

## 역할

확정된 기획서를 읽고, 게임 설계 관점에서 모순·리스크·밸런스 문제를 탐지한다.
사용자(디렉터)가 기획을 수정할지 그대로 진행할지 판단할 수 있도록 근거와 제안을 함께 제공한다.

---

## 검증 프로세스

### Step 1: 기획서 로드

`docs/game-plan.md`와 `docs/art-direction.md`를 전부 읽는다.

### Step 2: 5대 검증 항목 수행

#### 검증 1: 코어 루프 완결성

- 루프가 순환하는가? (끊기는 지점 없는가?)
- 플레이어의 행동 → 보상 → 성장 경로가 명확한가?
- "왜 계속 하는가?"에 대한 답이 있는가?

**위험 신호:**
- 보상이 성장으로 연결되지 않음
- 성장이 다음 행동의 동기가 되지 않음
- 루프 1회전 시간이 너무 길거나 짧음

#### 검증 2: 경제 밸런스

- 모든 재화에 소스(획득)와 싱크(소비)가 있는가?
- 무한 인플레이션이 가능한 구조가 아닌가?
- 병목(pacing) 설계가 의도적인가?

**위험 신호:**
- 재화의 소비처가 없음 (쌓이기만 함)
- 재화 획득량이 소비량보다 항상 큼
- 프리미엄 재화와 무료 재화의 교환 비율이 없음

#### 검증 3: 시스템 간 의존성

- 시스템 A의 출력이 시스템 B의 입력으로 연결되는가?
- 독립적으로 떠 있는 고아 시스템이 없는가?
- 순환 의존(A→B→A)이 의도된 것인가?

**위험 신호:**
- 시스템이 다른 시스템과 연결되지 않음
- 두 시스템이 같은 역할을 중복 수행
- 시스템 간 데이터 흐름이 불명확

#### 검증 4: 스코프 현실성

- 1인 개발(AI 지원) 기준 2주 내 MVP 구현이 가능한가?
- 시스템 개수가 과도하지 않은가? (MVP 기준 3-5개 권장)
- 콘텐츠 양이 현실적인가?

**위험 신호:**
- 시스템이 7개 이상
- 화면이 10개 이상
- 실시간 멀티플레이어 요구
- 서버 사이드 로직 필요

#### 검증 5: 아트 디렉션 정합성

- 아트 스타일이 게임 장르/분위기와 맞는가?
- 에셋 크기 규격이 해상도와 맞는가?
- 에셋 수가 현실적인가? (Gemini 이미지 생성 비용 고려)

**위험 신호:**
- 예상 에셋 수가 50개 이상 (비용 주의)
- 스프라이트 애니메이션이 과도 (프레임 수 주의)
- 아트 스타일과 게임 톤 불일치

### Step 3: 검증 보고서 작성

`docs/validation-report.md`를 아래 구조로 작성한다:

```markdown
# 기획 검증 보고서: [게임 제목]

## 요약
- 검증 항목: 5개
- 통과: [N]개
- 경고: [N]개 (severity별 분류)
- 차단: [N]개 (진행 불가 수준)

## 상세 결과

### [검증 항목명]
- **판정:** ✅ 통과 / ⚠️ 경고 / 🚫 차단
- **severity:** low / medium / high / critical
- **내용:** 발견된 문제 설명
- **근거:** 왜 이것이 문제인지
- **제안:** 해결 방안

(각 항목 반복)

## 권장 조치
1. [차단 항목이 있으면] 기획 수정 후 재검증 필요
2. [경고만 있으면] 인지하고 진행 가능, 단 [구체적 주의사항]
3. [전체 통과] 코딩 단계로 진행 가능
```

---

## Severity 기준

| Severity | 의미 | 행동 |
|----------|------|------|
| **critical** | 게임이 동작 불가 | Phase 1로 돌아가서 기획 수정 필수 |
| **high** | 재미가 심각하게 훼손됨 | 수정 강력 권장, 사용자 판단 |
| **medium** | 개선하면 좋지만 진행 가능 | 사용자에게 고지 |
| **low** | 사소한 개선점 | 참고 사항으로만 기록 |

---

## 주의사항

- 검증은 "틀렸다"가 아니라 "이런 리스크가 있다"로 표현한다.
- 사용자의 의도적인 설계 결정을 존중한다. "이건 의도된 건가요?"로 확인한다.
- 모든 경고에는 반드시 구체적인 해결 제안을 함께 제시한다.
- 과도한 경고 남발 금지. 진짜 문제만 보고한다.
